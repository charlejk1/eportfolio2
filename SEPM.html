Software Engineering Project Management

Unit 1 Introduction to Software Engineering Project Management

1.1 Software Engineering Life Cycle

Software Engineering Poject management requires working with internal and external stakeholders to provide a solution to a defined scope.

 

·      Understanding software development requirements

·      Deciding on resources required – internal and external.

·      Deciding on the Project Plan

·      Estimating time and cost

·      Stakeholder management

·      Risk Assessment and mitigation

·      Defining reporting requirements

·      Change management.

 

Management of Time, Scope and Cost affecting the quality of output.

 

Management of a Software engineering project requires an understanding of how to gather business requirements and deliver a quality product that meets the requirements. This must be achieved within defined timelines and budgets.

 

Project managers need the skill to pull together the required resources and manage any potential risks.

Unit 2 Why projects fail

2.1 Examples of Reasons for Project Failure

Table 1: Results from the interviews: reasons for software development process poor implementation. Marques et al (2017)

Some of the reasons for project failure are:

 

1. Unclear Software Engineering Project Requirements: This can be because of poorly documented or unconfirmed requirements which leads to a different interpretation of the scope among stakeholders.

 

Mitigation: Identify and clearly define the objectives of the project using techniques such as workshops and interviews. Stakeholders must sign off on the documented requirements.

 

2. Scope Creep: Implementation of undocumented scope

 

Mitigation: Define and educate stakeholders on the change control process. The introduction of a change approval process before any new requirements irrespective of assumed criticality needs to be put in place. Impact on the project's timeline, budget, and resources before approving them also needs to be carried out.

 

3. Poor Communication: No clear definition of reporting requirements

 

Mitigation: Implement a robust communication plan that defines communication channels, frequency, and responsibilities. Encourage open and transparent communication among team members and stakeholders.

 

4. Absence of or little Stakeholder Engagement:

 

Mitigation: Identify key stakeholders at the inception of the project and involve them in decision-making processes. This ensures that their needs and concerns are addressed throughout the project.

 

5. Mismanagement of Risks:

 

Mitigation: Identification and documentation of project risks early and regularly update the risk register. Develop risk response plans to address potential issues and uncertainties.

 

6. Insufficient Testing and Quality Control:

 

Mitigation: Implement a powerful testing strategy that includes unit testing, integration testing, and user acceptance testing. This ensures that a good quality control process is in place. It also allows for the resolution of the defect before releasing the developed or implemented software.

Unit 5

User experience

 

Usability:

 

Ease of use: How easily interact with the interface designed for user interfacing. This can be the user login page or access menu.

Easy to learn: The complexity of the product can affect user adoption.

Efficiency and Speed: The speed at which the programme processes the functions and accuracy.

Reliability: Ability to perform and the absence of crashes.

Relevance: Providing content that is useful and meaningful to the user.

Clarity: Ensuring that content is well-organized, concise, and easy to understand.

Structure: The arrangement of content and how users can navigate between sections or pages.

Searchability: The ability to find information quickly through effective search features.

Feedback: Providing users with clear feedback on their actions and the status of processes.

Error Handling: Guiding users when they make mistakes and helping them recover gracefully.

Tailoring the experience to user preferences and needs.

Ensuring that the product works well and is easy to use on various devices, including smartphones and tablets.

Multi-Browser Compatibility: Making sure the product works well on different web interfaces and software.

Emotional Impact: Designing for the emotional impact the product has on users, fostering positive emotions and engagement.

Unit 9

 

Quality Assessment

 

This allows for the development of a user-friendly system while maintaining the quality of the code. In the example below, the code allows for the code to be tested by capturing incorrect input such as password confirmation, user password and mobile number. It also protects the user’s data by preventing unintentional revealing of the user's data. The code also provides for confirmation of data entered correctly. In the example, the code’s storage option of a text file is identified as a weakness with the recommendation to use a database. The text file and encryption methods are identified as potential risks.

 

Code example

 

import hashlib

import re

 

def validate_mobile_number(mobile):

    """

    This function checks if the mobile number entered by the user is valid or not.

    """

    if re.match(r"^[6-9]\d{9}$", mobile):

        return True

    else:

        return False

 

def signup():

    mobile = input("Enter Mobile Number: ")

    if validate_mobile_number(mobile):

        pwd = input("Enter password: ")

        conf_pwd = input("Confirm password: ")

        if conf_pwd == pwd:

            # Hash the password using a more secure hashing algorithm like bcrypt

            hashed_pwd = hashlib.sha256(pwd.encode()).hexdigest()

 

            # Store the user's data in a database instead of a text file

            with open("credentials.txt", "a") as f:

                f.write(mobile + "," + hashed_pwd + "\n")

            print("You have registered successfully!")

        else:

            print("Passwords do not match. Please try again.")

    else:

        print("Incorrect mobile number format. Please enter a valid number.")

 

def login():

    mobile = input("Enter Mobile Number: ")

    pwd = input("Enter password: ")

    hashed_pwd = hashlib.sha256(pwd.encode()).hexdigest()  # Hash the entered password

 

    # Retrieve user credentials from a database instead of reading from a text file

    with open("credentials.txt", "r") as f:

        lines = f.readlines()

    for line in lines:

        stored_mobile, stored_pwd = line.strip().split(",")

        if mobile == stored_mobile and hashed_pwd == stored_pwd:

            print("Logged in Successfully!")

            break

    else:

        print("Login failed. Please check your mobile number and password.")

 

while True:

    print("********** Login System **********")

    print("1.Signup")

    print("2.Login")

    print("3.Exit")

    ch = int(input("Enter your choice: "))

    if ch == 1:

        signup()

    elif ch == 2:

        login()

    elif ch == 3:

        break

    else:

        print("Incorrect Selection!")

MaCall et al., 1977 model focuses on three categories of a product, transition, operation, and revision.  The model identifies the need for requirements to be gathered accurately to deliver a good product. This also highlights the ability to reuse and upgrade the code. According to the model, the product must provide the ability to test without impacting on flexibility.

 

In a more recent article by Kokol, P (2022), adds big data, data mining, indexing and artificial intelligence to the points in the MaCall et al., 1977 model. This allows for the better usage of the data that is gathered during the process. The use of artificial intelligence allows for the better modelling of the data. Kokol still maintains the need for the items in the model tree as being a requirement in more recent model.
